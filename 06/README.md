# 6주차 - RAG(Retrieval-Augmented Generation) 아키텍처 적용

## 개요

5주차에서 직접 구현한 MiniGPT는 "다음 단어 예측"만 학습된 모델이라, 질문을 던지면 의도를 이해하지 못하고 특정 단어를 반복하는 한계가 있었다. 이를 보완하기 위해 6주차에서는 **RAG(검색 증강 생성)** 아키텍처를 적용했다.

RAG는 질문이 들어오면 (1) 관련 문서를 미리 구축한 벡터 저장소에서 검색하고, (2) 검색된 문서를 근거로 LLM이 답변을 생성하는 구조다. 모델을 다시 학습시키는 대신, "참고 자료를 찾아서 함께 물어보는" 방식으로 정확도와 신뢰도를 높이는 접근이다.

### 왜 답변 생성에 MiniGPT 대신 Gemini 2.5 Flash를 사용했는가

5주차 MiniGPT는 "다음 단어 예측"만 학습되어 질문-답변 형태의 출력을 만들 수 없다는 한계가 명확했다. RAG의 핵심은 "검색된 문서를 이해하고 질문에 맞게 요약/추출하는 능력"인데, 이는 사전학습된 LLM의 언어 이해 능력이 필요한 영역이다. 따라서 6주차에서는 **검색(임베딩)은 로컬 모델, 답변 생성은 Gemini 2.5 Flash API**로 역할을 분리해 구현했다.

> MiniGPT 자체를 RAG의 답변 생성기로 활용하는 것은 별도 과제로 분류했다 (아래 [한계점 및 향후 방향](#한계점-및-향후-방향) 참고).

## 아키텍처

```
[사용자 질문]
      │
      ▼
┌─────────────────┐
│   retriever.py   │  질문을 벡터로 변환 (BAAI/bge-m3)
│  (검색 담당)      │  FAISS 벡터 저장소에서 유사 문서 top-3 검색
└─────────────────┘
      │  검색된 문서 3개
      ▼
┌─────────────────┐
│  generator.py    │  질문 + 검색된 문서를 프롬프트로 구성
│  (답변 생성 담당) │  Gemini 2.5 Flash API 호출
└─────────────────┘
      │
      ▼
┌─────────────────┐
│ rag_pipeline.py  │  retriever + generator 통합
└─────────────────┘
      │
      ▼
┌─────────────────┐
│    main.py       │  FastAPI로 REST API 서빙 (POST /ask, 포트 8001)
└─────────────────┘
      │
      ▼
[answer + sources 반환]
```

## 기술 스택

| 구분 | 사용 기술 |
|---|---|
| 검색 대상 데이터 | KorQuAD v1.0 (지문 9,681개) |
| 청킹 | LangChain `RecursiveCharacterTextSplitter` (chunk_size=500, overlap=50) → 13,842개 청크 |
| 임베딩 모델 | BAAI/bge-m3 (로컬 실행, Colab A100 GPU) |
| 벡터 저장소 | FAISS |
| 답변 생성 LLM | Gemini 2.5 Flash API |
| API 서버 | FastAPI (POST `/ask`, 포트 8001) |
| API 키 관리 | `.env` + `python-dotenv` |

## 테스트 결과

### ✅ 성공 사례 (검색 대상 문서 내에 답이 존재하는 질문)

```
Q: 이순신의 장인은 누구인가?
A: 이순신의 장인은 보성군수를 역임한 방진(方震)이다.
```

검색된 문서(source) 내에 답변 근거 문장이 그대로 포함되어 있으며, 환각(hallucination) 없이 정확하게 답변을 생성했다.

### ⚠️ 한계 사례 (검색 대상 문서 내에 답이 없는 질문)

```
Q: 오늘 날씨에 대해 알려줘
A: 제공된 문서에는 오늘 날씨에 대한 정보는 없습니다.
```

문서 내에 관련 정보가 없을 때, 근거 없는 답을 지어내지 않고 "정보 없음"으로 정직하게 응답하는 것을 확인했다. 다만 이 경우에도 retriever는 질문과 실제로는 무관한 문서 3개를 검색 결과로 반환했다 (아래 한계점 참고).

## 한계점 및 향후 방향

1. **retriever가 관련성이 낮아도 top-k(3개) 문서를 강제로 반환하는 구조적 한계**
   질문과 검색 대상 문서 사이에 실질적인 연관성이 없어도, FAISS는 상대적으로 거리가 가까운 순서로 무조건 3개를 반환한다. 이번 테스트에서는 답변 생성 단계(Gemini)가 이를 걸러내 정직하게 "정보 없음"으로 응답했지만, 검색 단계 자체의 정확도를 높일 필요가 있다.
   → 향후 최소 유사도 임계값(threshold)을 추가해, 일정 기준 이하로 관련성이 낮은 문서는 아예 검색 결과에서 제외하는 방식으로 개선 예정.

2. **검색 대상 데이터(KorQuAD)의 주제 범위가 제한적**
   KorQuAD는 9,681개 지문으로 구성된 상대적으로 작은 데이터셋이라, 다룰 수 있는 질문의 폭이 넓지 않다.
   → 향후 AI 허브(aihub.or.kr) 등 공개 데이터셋을 추가해 검색 대상 데이터의 다양성을 확대할 예정.

3. **MiniGPT(자체 모델) 기반 RAG는 별도 과제로 분리**
   MiniGPT는 다음 단어 예측만 학습되어 질문에 답하는 능력이 없다. 자체 모델 기반 RAG를 만들려면 검색된 문서에서 정답 구간을 그대로 찾아 반환하는 추출형(extractive) QA 방식이 필요하며, 이는 MiniGPT에 별도의 "정답 위치 분류" head를 추가하는 학습이 요구된다. 7주차 상담 준비(자체 모델 vs Gemma vs Gemini 비교 평가) 과정에서 시간을 두고 진행할 예정이다.

## 파일 구조

```
06/
├── retriever.py        # FAISS 검색 (BAAI/bge-m3 임베딩, load_retriever(), search())
├── generator.py         # Gemini 2.5 Flash 답변 생성 (configure_gemini(), build_prompt(), generate_answer())
├── rag_pipeline.py      # retriever + generator 통합 (init_pipeline(), ask())
├── main.py               # FastAPI 서빙 (POST /ask, 포트 8001)
├── faiss_index_local/   # FAISS 벡터 저장소 (index.faiss, index.pkl)
├── .env                 # GEMINI_API_KEY 저장 (.gitignore 처리)
└── README.md
```

## 실행 방법

### 1. 필요 라이브러리 설치

```bash
pip3 install fastapi uvicorn python-dotenv google-generativeai langchain faiss-cpu sentence-transformers
```

### 2. `.env` 파일 생성

`06/` 폴더 안에 `.env` 파일을 만들고 아래와 같이 본인의 Gemini API 키를 입력한다.

```
GEMINI_API_KEY=발급받은_API_키
```

### 3. 서버 실행

```bash
cd 06
python3 main.py
```

아래와 같은 로그가 출력되면 정상적으로 실행된 것이다.

```
RAG 파이프라인 초기화 완료!
INFO:     Uvicorn running on http://0.0.0.0:8001 (Press CTRL+C to quit)
```

### 4. API 테스트

브라우저에서 아래 주소로 접속하면 Swagger UI를 통해 직접 테스트할 수 있다.

```
http://localhost:8001/docs
```

`POST /ask`에 아래와 같은 형식으로 질문을 입력한다.

```json
{
  "question": "이순신의 장인은 누구인가?"
}
```

## 회고

5주차에서는 모델을 처음부터 직접 학습시키는 경험을 했다면, 6주차에서는 "이미 있는 강력한 언어모델을 어떻게 신뢰도 높게 활용할 것인가"에 초점을 맞췄다. 특히 Gemini 임베딩 API의 요청 한도에 막혀 로컬 임베딩 모델로 전환한 과정에서, 클라우드 API에만 의존하지 않고 로컬 리소스(Colab GPU)를 적절히 활용하는 판단을 스스로 내려본 것이 의미 있었다.

또한 테스트 과정에서 retriever가 관련 없는 질문에도 문서를 강제로 반환하는 구조적 한계를 직접 발견했는데, 이는 RAG 시스템에서 "검색 정확도"와 "생성 단계의 안전장치"가 별개로 설계되어야 한다는 점을 체감하는 계기가 되었다. 이 한계점은 7주차 이후 개선 과제로 이어갈 예정이다.
